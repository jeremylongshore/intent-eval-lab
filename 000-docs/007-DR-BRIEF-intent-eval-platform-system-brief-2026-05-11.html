Intent Eval Platform
How the System Works: A Complete Architecture Brief

This document describes the Intent Eval Platform — a three-repository evaluation system being built by Intent Solutions — from the ground up. It covers what the platform is, why it exists, how each component works, how the pieces connect, and what the full journey looks like from a developer shipping code to a signed, independently verifiable evaluation result. If you read one document to understand this system, this is it.

1. The Problem This Solves

The AI tooling ecosystem has a measurement problem. As AI agents, MCP servers, SKILL.md files, and CLI tools have proliferated across the industry, nobody has built a standardized way to answer the question: did this thing actually work correctly?

Every vendor has their own eval. Every team has their own benchmark. Claude Code has its own tests, Cursor has its own, Cline has its own, and none of them speak the same language. When a developer plugs an MCP server into Claude Code and something goes wrong — a tool call races, a cooldown isn't respected, a side effect fires twice — there is no standard format for describing that failure, recording it, or verifying that a fix actually resolved it.

The situation is worse on the MCP server side specifically. MCP is an open protocol that allows AI agents to call external tools. As of today, there is no benchmark, no conformance standard, and no independently verifiable attestation format for MCP servers. A vendor can claim their MCP server "works with Claude Code" and there is no standard way to prove or disprove that claim.

The Intent Eval Platform is the answer to that gap. It is being built as candidate substrate for an industry standard — not internal tooling, not a product with a single vendor's logo on it, but a set of specifications, schemas, and open-source tools that any party can use to produce independently verifiable evaluation results.

The core principle: Every evaluation result should be expressible as a signed, independently verifiable artifact that any auditor can check without trusting the team that produced it. Partial results are valid. Silence about a dimension means "not tested," not "failed." Coverage is a derived signal, not a requirement.

2. The Three Repositories

The platform is composed of three independent open-source repositories. Each has its own Git history, its own release cycle, its own license, and its own package. They are not a monorepo. They compose at the schema layer — they all emit the same kind of result artifact — and that is the only required coupling between them.

RepositoryRoleLicenseLanguage
intent-eval-labMethodology + specs + schema definitions. The rulebook everything else follows.Apache 2.0Markdown / JSON Schema
audit-harnessStatic gate runner. Checks the code and configuration of a repository without running a model.MITBash + Python + Node.js dispatcher
j-rig-binary-evalBehavioral gate runner. Drives an AI model through a task and judges whether it behaved correctly.MITTypeScript (pnpm monorepo)

The local filesystem lays them out as siblings under a shared umbrella directory at ~/000-projects/intent-eval-platform/. That directory is a filesystem grouping — it has no build system, no shared package.json, and no git history of its own. The coupling is entirely at the schema layer.

3. The Five Surfaces

The platform is designed to evaluate five distinct surfaces. Each surface is independently evaluable. A team that only ships MCP servers does not need to evaluate SKILL.md files. A team that only ships CLI tools does not need to run behavioral MCP gate checks. The system is designed so that partial coverage is explicitly valid — and so that the evaluation result clearly states which surfaces were tested and which were not.

SurfaceWhat It IsExamples
APIA provider-side HTTP or RPC endpoint that returns model output or tool resultsAnthropic Messages API, OpenAI Responses API, MCP server tools/call JSON-RPC
CLIThe local binary an operator runs to invoke a model or agentclaude, codex, gemini, cursor, aider, cline
MCP ServerA separately-running protocol surface the agent invokes via JSON-RPCA Kobiton test automation server, a Plane project management server, any custom domain MCP server
AgentThe runtime that composes API calls, tool calls, memory, and hooksClaude Code, Codex CLI, Gemini CLI, Cline, Cursor, Continue.dev
SKILL.mdThe instruction-shaped artifact that primes an agent for a specific taskSKILL.md, AGENTS.md, .cursor/rules/*.mdc, .windsurf/rules/*.md, GEMINI.md

The transformation pipeline that connects these surfaces is: API → CLI → MCP server → SKILL.md → agent runtime. A capability or failure can be traced along any hop in that chain. When the platform attests to a result, the subject of the attestation names which hop is being described — so a consumer can know whether a claim is about the server side, the client side, or the instruction layer.

4. The Audit Harness — Static Gates

The audit harness is a static analysis tool. It runs against a repository without invoking any AI model. Its job is to check whether the code, configuration, and test artifacts of a repository meet a defined quality bar. Think of it as a highly opinionated linter that knows about AI-specific failure modes.

The harness is published as @intentsolutions/audit-harness on npm and is also available as a PyPI package, a Rust crate, and via a curl install script. The primary interface is a Node.js CLI dispatcher (audit-harness) that routes to six underlying scripts.

The six gates

GateWhat It ChecksOutput
escape-scanScans staged git diffs for AI escape patterns — specifically REFUSE, CHALLENGE, and FLAG grammar that indicates an AI model attempting to exit a constrained task. Thresholds are read from tests/TESTING.md so enforcement travels with the code.PASS / FAIL + matching lines
crap-scoreComputes the CRAP metric (Change Risk Anti-Patterns) — a function of cyclomatic complexity multiplied by the inverse of test coverage. High CRAP score = high-risk code. Supports Python, JavaScript, Go, Rust, Java, PHP, Ruby, and .NET.PASS / FAIL + per-function scores
arch-checkRuns dependency-cruiser to enforce architectural rules — which modules are allowed to import which other modules. Prevents circular dependencies and layer violations.PASS / FAIL + violation list
harness-hashMaintains a SHA-256 manifest of policy files (.feature files, dependency-cruiser config, ArchUnit config, Stryker config). Detects tampering — if a policy file changes without the hash being re-initialized, the gate fails. Prevents AI-proposed policy changes from silently bypassing enforcement.PASS / FAIL + hash diff
bias-countCounts test-bias patterns in the test suite — tautological assertions, tests that always pass, mocked implementations of the function under test. Advisory output.COUNT + flagged tests (advisory)
gherkin-lintLints Gherkin feature files for structural quality — step definition hygiene, scenario naming, tag discipline.PASS / FAIL + lint violations (advisory)

How the harness runs today

Today the harness produces text output and exit codes. Exit code 0 means pass, 1 means fail, 2 means error. The results are human-readable. They work in CI pipelines, pre-commit hooks, and local development workflows. When you run audit-harness escape-scan --staged in a git hook, you get a human-readable report telling you whether any staged AI-generated diffs contain escape patterns.

The crap-score gate already supports a --format json flag that emits machine-readable output. That flag is the reference implementation for the Phase B upgrade path — every other gate will get the same treatment.

The upgrade path (Phase B)

The Phase B upgrade adds two capabilities on top of the existing gates without changing any gate logic:

AH-2 — --json flag on all subcommands. Every gate will accept a --json flag that causes it to emit structured JSON instead of human-readable text. The gate logic is unchanged. The output format gains a machine-readable mode. This is approximately 30 lines of code per script, following the crap-score pattern that already exists.

AH-3 — emit-evidence subcommand. A new subcommand that takes the JSON output from any gate, wraps it in the Evidence Bundle row format (an in-toto Statement v1), and writes the signed row to disk or stdout. This is what connects the static gate to the shared evaluation standard. Once a gate result is in Evidence Bundle format, it is independently verifiable by any party with access to the signing key's public component.

Both upgrades are additive. The existing text-mode behavior is unchanged. Developers who use the harness for their own repos today will notice nothing different unless they opt into --json mode.

5. J-Rig Binary Eval — Behavioral Gates

J-rig is the behavioral evaluation harness. Where audit-harness checks static properties of a codebase, j-rig checks dynamic properties of an AI system in action. It drives a model through a task, observes what the model does, and judges whether the behavior matches a specification.

J-rig is a TypeScript pnpm monorepo with four workspace packages: core (schemas and evaluation engine), CLI (command-line interface), database (result persistence), and dashboard (reporting). The evaluation engine reads an eval spec — which can be a SKILL.md file, an AGENTS.md file, or a custom eval definition — and produces a structured judgment result.

The Intentional Mapping — MM-1 through MM-6

The most important concept in j-rig is the Intentional Mapping. This is a taxonomy of failure shapes that were discovered in real partner engagements — not theorized, not invented from survey data, but found in actual production failures observed during Intent Solutions' multi-vendor work. There are currently six categories, designated MM-1 through MM-6. The numeric identifiers are permanent; they will not be renamed even as the conceptual vocabulary evolves around them.

IDCategoryWhat It Catches
MM-1Async RaceTool calls that fire before a prior asynchronous operation has settled. The agent invokes Tool B before Tool A's result is available, leading to incorrect state or silent data loss.
MM-2Shape DriftThe response shape of a tool call diverges from what the SKILL.md or AGENTS.md declared. The agent's declared interface and its actual runtime interface have diverged.
MM-3CooldownRate-limit or cooldown semantics are not respected. The agent calls the same tool again before the declared cooldown period has elapsed, causing server-side rate-limit failures or silent throttling.
MM-4Side-Effect VerificationA tool call produces a side effect (writes a file, sends a message, modifies state) that is never verified by a subsequent observation. The agent assumes success without confirming it.
MM-5Mandatory ContextThe agent invokes a tool without providing context that the tool's interface requires. The tool call is technically valid but semantically incomplete — the server cannot give a useful response.
MM-6Protocol Strict-ModeThe MCP server operates in strict protocol mode but the agent sends a request that would only be valid in lenient mode. Strict-mode rejections are not surfaced to the user.

These six categories are expressed as Gherkin scenarios in the platform's spec directory. Any party can read them, implement their own test runner against them, and contribute results to the Evidence Bundle format. The goal is for MM-1 through MM-6 to become a community standard — filed as part of an OpenTelemetry semantic conventions RFC — so that every tool vendor, every agent runtime, and every MCP server can describe their conformance (or non-conformance) to these same six categories.

What j-rig does during an evaluation

A j-rig evaluation run follows this sequence:

  1. Parse the eval spec. J-rig reads the SKILL.md file (or AGENTS.md, or a custom eval definition). It extracts the declared tool interfaces, allowed-tools list, effort level, and any explicit acceptance criteria.
  2. Drive the model. J-rig invokes the model against the eval task. In Phase B, this will be done through provider adapters (either LiteLLM or the Vercel AI SDK — that decision is deferred to in-prototype measurement). The provider adapter handles the actual API calls; j-rig observes the inputs and outputs.
  3. Observe behavior. J-rig records every tool call, every response, every state transition. OpenTelemetry spans fire during this phase — specifically the agent.rollout.gate.* events that the Intent Eval Platform is proposing as a new semantic convention.
  4. Judge against the Intentional Mapping. J-rig checks the recorded behavior against the MM-1 through MM-6 categories. Each check produces a PASS, FAIL, or NOT-APPLICABLE verdict.
  5. Emit Evidence Bundle rows. Each verdict becomes an in-toto Statement v1 row, signed with the evaluation operator's key, and added to the Evidence Bundle for this run.

6. The Evidence Bundle — The Shared Output Format

The Evidence Bundle is the single most important concept in the platform. It is the shared output format that allows audit-harness and j-rig to compose without knowing about each other, and that allows any external party to verify evaluation results without trusting the team that produced them.

An Evidence Bundle is a collection of in-toto Statement v1 rows. Each row is an independently signed attestation about one specific gate result at one specific moment in time. A bundle can contain one row or one hundred rows. Partial bundles are explicitly valid — they mean "we tested these things and did not test those things," not "we failed everything we did not test."

What a single Evidence Bundle row looks like

{
  "_type": "https://in-toto.io/Statement/v1",
  "subject": [
    {
      "name": "audit-harness:client:escape-scan",
      "digest": {
        "sha256": "abc123def456..."
      }
    }
  ],
  "predicateType": "https://evals.intentsolutions.io/gate-result/v1",
  "predicate": {
    "gate_id": "audit-harness:client:escape-scan",
    "result": "PASS",
    "policy_hash": "sha256:789abc...",
    "input_hash": "sha256:def012...",
    "timestamp": "2026-05-11T14:23:00Z",
    "runner": "audit-harness@0.3.0",
    "commit_sha": "9971183"
  }
}

A few things to notice about this structure. The subject field uses the in-toto Statement standard's subject naming to identify the pipeline hop being attested — in this case, the client-side escape-scan gate. The predicateType is a URI hosted at evals.intentsolutions.io, which is the Intent Solutions predicate URI namespace locked by the ISEDC council decision on 2026-05-10. The predicate contains the actual gate result data in the Intent Solutions gate-result schema.

Once a row is signed and pushed to the Rekor transparency log, it is permanently recorded. Any party with network access to Rekor can verify the signature, confirm the commit SHA, and independently reconstruct the gate verdict. The evaluation result is no longer dependent on trusting Intent Solutions — it is anchored in a public, append-only transparency log.

Why in-toto Statement v1

In-toto is a supply chain security framework originally developed at NYU and now a CNCF graduated project. Its Statement v1 format is the same envelope used by SLSA provenance attestations, SCAI (Supply Chain Attribute Integrity) statements, and Cosign-signed image manifests. By adopting in-toto Statement v1 as the Evidence Bundle envelope, the platform gets several things for free:

  • GUAC ingestion. GUAC (Graph for Understanding Artifact Composition) can ingest in-toto Statements natively. Any organization already running GUAC for supply chain visibility can ingest Intent Eval Platform results without additional integration work.
  • Cosign compatibility. Cosign signs in-toto Statements. The signing infrastructure is already built and battle-tested.
  • Rekor transparency log. Rekor accepts in-toto Statements. Every signed evaluation result gets a permanent, timestamped, publicly verifiable entry.
  • Composability. The in-toto Statement's subject field naturally carries the pipeline-hop discriminator — which surface, which side (client or server), which hop in the transformation chain. No bespoke field is needed.

7. The OpenTelemetry Layer

While the Evidence Bundle carries the durable, signed record of evaluation results, OpenTelemetry carries the ephemeral, real-time observability signals that fire during an evaluation run. These two layers are complementary: OTel is for watching what happens as it happens; the Evidence Bundle is the permanent record of what happened.

The Intent Eval Platform is proposing two new semantic convention groups for the OpenTelemetry community:

agent.rollout.gate.* events. These events fire when a CI rollout gate evaluates — when audit-harness runs escape-scan, when j-rig produces a verdict, when the Rollout Gate makes a ship/no-ship decision. The events carry attributes like gate.id, gate.result, gate.policy_hash, and gate.commit_sha. Any OpenTelemetry collector will capture these events alongside normal application telemetry, giving operators a single observability plane that spans code quality gates and behavioral evaluation gates.

Intentional Mapping signal vocabulary. The MM-1 through MM-6 category names are being proposed as named OTel semantic conventions. Today they are markdown prose — "MM-3 cooldown" is a human-readable label. Once standardized as OTel conventions, they become filterable signals: any collector, any dashboard, any alert rule can filter on agent.mm_category == "MM-3" without parsing custom log formats.

The OTel RFC for agent.rollout.gate.* is already drafted at intent-eval-lab/000-docs/001-DR-RFC-otel-agent-rollout-gate-signals-draft.md. The community-engagement process for filing this RFC with the OTel SIG-GenAI working group follows the Intent Solutions CSO sequence: informal community-temperature email to a verified SIG-GenAI maintainer first, RFC filing four or more weeks later informed by the routing feedback.

8. The Rollout Gate

The Rollout Gate is a GitHub Action that runs at the end of a CI pipeline. It reads the Evidence Bundle produced by audit-harness and j-rig during the pipeline, evaluates the coverage and pass rates against a declared policy, and makes a ship/no-ship decision. This is the Phase C deliverable — it is the piece that closes the loop from evaluation to deployment decision.

The policy that the Rollout Gate enforces is declared in the repository's tests/TESTING.md file — the same file that audit-harness reads for its thresholds. The enforcement travels with the code. Any developer who clones the repository and runs the CI pipeline gets the same gate with the same thresholds. There is no external service to configure, no SaaS dashboard to log into, no vendor-specific configuration format to learn.

The Rollout Gate does not require complete coverage. A repository that covers three of the six MM categories and two of the five surfaces can still pass the Rollout Gate if its declared policy only requires those three categories and two surfaces. The composable partial attestation principle means the gate is as strict or as lenient as the team declares in their policy — and the Evidence Bundle clearly shows which dimensions were covered and which were not.

9. How It All Works Together — The Full Journey

Here is the complete journey from a developer pushing code to a signed, independently verifiable evaluation result sitting in the Rekor transparency log.

Step 1 — Developer pushes a commit

A developer on a team that uses the Intent Eval Platform pushes a commit to their repository. The commit might be a change to a SKILL.md file, a new MCP server endpoint, a modification to a Claude hook, or an update to the agent runtime configuration.

Step 2 — Pre-commit hooks fire

Before the commit is accepted, git pre-commit hooks run audit-harness in static gate mode. The escape-scan gate checks the staged diff for AI escape patterns. The harness-hash gate verifies that no policy files have been modified without re-initializing the hash manifest. If either gate fails, the commit is blocked and the developer sees a human-readable report describing what needs to be fixed.

Step 3 — CI pipeline begins

The commit reaches the CI pipeline. The pipeline runs the full audit-harness suite — all six gates — against the repository state at that commit. Each gate runs with the --json flag (Phase B upgrade AH-2), producing structured JSON output in addition to the human-readable text.

Step 4 — Static gate results emitted as Evidence Bundle rows

The emit-evidence subcommand (Phase B upgrade AH-3) runs after each gate. It takes the JSON output, wraps it in an in-toto Statement v1 using the evals.intentsolutions.io/gate-result/v1 predicate type, signs it with the operator's key, and writes the signed row to the Evidence Bundle for this run. At this point the bundle contains rows for all six static gates.

Step 5 — Behavioral evaluation runs

J-rig begins its evaluation run. It parses the repository's SKILL.md or AGENTS.md file, extracts the declared tool interfaces and behavioral expectations, and drives the model through the evaluation task using the configured provider adapter. OpenTelemetry spans fire throughout this process — every gate evaluation, every tool call, every rollout decision emits an agent.rollout.gate.* event that is captured by whatever OTel collector the operator has configured.

Step 6 — Intentional Mapping checks run

J-rig checks the observed behavior against the MM-1 through MM-6 categories. For each applicable category, j-rig records a PASS, FAIL, or NOT-APPLICABLE verdict. A repository that does not ship an MCP server will receive NOT-APPLICABLE verdicts for the MCP-specific checks (MM-3, MM-6) — this is not a failure. It is an accurate statement about which surfaces were evaluated.

Step 7 — Behavioral gate results emitted as Evidence Bundle rows

Each j-rig verdict becomes an in-toto Statement v1 row, signed with the same operator key, and added to the Evidence Bundle. The bundle now contains both static gate rows (from audit-harness) and behavioral gate rows (from j-rig). The rows are independent — neither tool needed to know about the other to produce its rows, and the rows can be verified independently by any party.

Step 8 — Rollout Gate evaluates the bundle

The Rollout Gate GitHub Action reads the complete Evidence Bundle and evaluates it against the repository's declared policy in tests/TESTING.md. It checks pass rates, coverage dimensions, and any specific requirements declared in the policy. If the bundle meets the policy, the gate passes and the CI pipeline continues to deployment. If the bundle fails the policy, the gate fails, the deployment is blocked, and the developer receives a report showing which rows failed and which were not covered.

Step 9 — Bundle is archived

Regardless of the gate outcome, the signed Evidence Bundle is archived. The rows are pushed to the Rekor transparency log. The evaluation result is now permanently recorded — timestamped, signed, publicly verifiable, immutable. A partner, an auditor, or a standards-body reviewer can retrieve the bundle from Rekor at any future point and independently verify every row without contacting Intent Solutions.

10. How Each Surface Is Covered

The diagram above shows the journey for a single surface (SKILL.md + agent behavioral eval). Here is how the system covers each of the five surfaces.

SKILL.md and AGENTS.md

Audit-harness checks SKILL.md files statically: the harness-hash gate pins the policy contents so they cannot be silently modified; the escape-scan gate checks any SKILL.md-adjacent staged content for escape patterns. J-rig evaluates SKILL.md files behaviorally: it parses the file, extracts the declared interface, and drives a model through the task described in the skill, judging whether the model's behavior matches the SKILL.md's intent. A signed Evidence Bundle row attests to each judgment.

MCP Servers

MCP server evaluation is primarily a j-rig concern. J-rig's MM-1 through MM-6 behavioral categories were derived directly from MCP server failure patterns observed in real partner engagements. When j-rig evaluates an MCP server, it drives an AI agent to call the server's tools and observes whether the server's behavior matches its declared protocol contract. The six MM categories cover the full space of failure shapes that have been empirically observed: async races, shape drifts, cooldown violations, side-effect verification failures, missing-context invocations, and strict-mode protocol violations. Phase B will also include a Pact-style contract format for MCP server conformance, allowing server authors and agent-harness authors to share formal contracts via existing Pact broker infrastructure.

Claude Code Hooks

Claude hooks are the pre/post-tool-call intercept points that Claude Code exposes. The audit-harness escape-scan gate is specifically designed to audit hook-adjacent code — it scans git diffs for the REFUSE, CHALLENGE, and FLAG patterns that a hook might emit in response to AI escape attempts. Running escape-scan as a pre-commit hook means that every commit to a repository that ships hook configuration is automatically checked for hook quality before it reaches the main branch.

Agents

The agent surface is where j-rig does its primary work. The evaluation task drives the agent — Claude Code, Codex CLI, or another runtime — and observes its behavior via OTel spans. The agent.rollout.gate.* events fire from within the agent runtime during evaluation, providing a real-time stream of what the agent is doing. J-rig's behavioral verdict is based on what the OTel spans reveal about the agent's decision-making: did it respect the declared tool allowlist? Did it handle the cooldown correctly? Did it verify side effects?

APIs

The API surface is covered by j-rig's provider adapters (Phase B work item PB-7). A provider adapter wraps a specific model provider's API — Anthropic's Messages API, OpenAI's Responses API, Google's Gemini API — behind a common Provider interface. This allows j-rig to drive any supported model provider through the same evaluation task and compare results across providers. The adapter handles the API call; j-rig observes the call and response; the Evidence Bundle row attests to which provider, which model version, and what the result was.

11. The Standards Substrate Vision

Everything described above is being built to be consumed by parties outside of Intent Solutions — by partners, by standards bodies, by any development team that wants independently verifiable evaluation results for their AI systems.

The in-toto predicate URI at evals.intentsolutions.io/gate-result/v1 is being registered as a public predicate type. This means any tool — not just audit-harness or j-rig — can produce rows in this format and contribute them to an Evidence Bundle. A team using a different static analysis tool can emit rows with the same predicate type. A team that builds their own MCP server conformance checker can emit rows with the same predicate type. The format is the standard; the tools are reference implementations of that standard.

The OTel semantic conventions being proposed — agent.rollout.gate.* events and the Intentional Mapping signal vocabulary — are being filed with the OpenTelemetry SIG-GenAI working group as community proposals. If adopted, they will become part of the shared observability vocabulary that every vendor in the AI tooling space can use, regardless of which evaluation tool they use to produce results.

The Intentional Mapping categories themselves — MM-1 through MM-6 — are published under Apache 2.0 in the intent-eval-lab repository. Any organization can contribute a new failure-shape category by observing a real production failure that does not fit the existing six categories and submitting it through the community contribution process at specs/CONTRIBUTING-failure-shape.md. The taxonomy grows from empirical evidence, not from theoretical survey — and the community contribution path is the mechanism that keeps it that way over time.

The distinguishing principle: The platform is built to be an industry standard, not a product. A standard outlives any single implementation. The Evidence Bundle schema, the Intentional Mapping categories, and the OTel semantic conventions are designed to remain useful and valid even if Intent Solutions never ships another line of code against them. That is the property that makes them worth building.

13. Summary

The Intent Eval Platform is three open-source repositories composing into one evaluation system via a shared schema. Audit-harness runs static gates against code repositories. J-rig runs behavioral gates against AI systems in action. Intent-eval-lab defines the standards — the Evidence Bundle schema, the Intentional Mapping failure-shape taxonomy, and the OpenTelemetry semantic conventions — that the other two emit against.

The output of the system is an Evidence Bundle: a collection of independently signed in-toto Statement v1 rows, each attesting to one gate result at one moment in time. The bundle is composable — partial coverage is valid — and independently verifiable by any party with Rekor access. No trust in Intent Solutions is required to verify an Evidence Bundle result.

The system covers five surfaces: APIs, CLIs, MCP servers, AI agents, and SKILL.md instruction files. It covers both the client side (what the agent runtime does) and the server side (what the MCP server or API enforces). Every surface, every side, every pipeline hop is independently attestable — and partial attestation is explicitly valid.

The build sequence starts with PB-1: the in-toto Statement v1 envelope schema. That is the first domino. Everything else follows from it.

The addendum below records the fourth repository's creation per Milestone 4 of the build journey plan (~/.claude/plans/se-the-council-bubbly-frog.md) and the gap analysis at 008-DR-GAPS-spec-vs-system-brief-2026-05-11.md.

What changed

  • New repository: jeremylongshore/intent-rollout-gate created 2026-05-12 as a public MIT-licensed repo. Local clone at intent-eval-platform/intent-rollout-gate/, sibling to the other three convergence repos.
  • What it ships at M4: a substantive bootstrap — action.yml with declared inputs (bundle-path, policy-file, predicate-uri, optional rekor-url, cosign-key, dry-run) and outputs (decision, summary, signed-decision-row-path); a CI smoke that asserts the early-adopter contract (decision=not-implemented exit-0); a substantive design doc at 000-docs/001-DR-DESIGN-rollout-gate-architecture-2026-05-12.md; standard governance files (README, LICENSE, CONTRIBUTING, CLAUDE, SECURITY); a minimal CI workflow; and a beads baseline.
  • What it does NOT yet ship: the actual Rollout Gate decision logic. That lands in M5 per the build journey plan. The action's runs.using: composite stub returns decision=not-implemented with exit 0 by design — this lets adopters wire uses: jeremylongshore/intent-rollout-gate@v1 into their CI ahead of M5 without the action blocking their deployments.

Why a fourth repo (not a fourth module in intent-eval-lab)

The Rollout Gate is a downstream consumer of the Evidence Bundle, not a producer of methodology or specs. Authoring it as a module inside intent-eval-lab would conflate the lab's role (defining the standard) with the gate's role (acting on the standard). It would also force every gate-implementation change through a methodology-repo PR cycle, which is the wrong governance shape — gate decisions are operational, not normative.

The four-repo split mirrors the Evidence Bundle architecture itself: intent-eval-lab defines the predicate URI; audit-harness + j-rig-binary-eval emit rows against it; intent-rollout-gate consumes assembled bundles + policy. Each repo has one job. None depends on shipping the others' code; they depend only on the schema.

What this means for § 8 of the original brief

§ 8 ("The Rollout Gate") describes the architectural intent. It remains accurate as the design narrative. The implementation lives at the URL above, not as a module inside one of the other repos. The design doc at intent-rollout-gate/000-docs/001-DR-DESIGN-rollout-gate-architecture-2026-05-12.md extends § 8 with the three-stage decision algorithm (required-gate pass + applicable-only coverage + advisory elevation), the OTel events fired during evaluation, the language-choice deferred decision (TS vs Go vs Python — locked in M5), and the early-adopter exit-0 contract.

What this means for the build sequence

§ "Phase B build sequence" remains correct: PB-1 (in-toto envelope schema) is still the first domino. The fourth repo's existence does not reorder PB-1..PB-12 because the fourth repo's only dependency is the schema PB-1 ships. The rollout-gate implementation work is M5 in the build journey plan, gated on M3 completion (j-rig emission) — same gating as before; only the repo it lives in changed.
