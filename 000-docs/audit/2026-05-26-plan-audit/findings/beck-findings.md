# Kent Beck — Plan Audit Findings — 2026-05-27

## Summary (top concerns)

- **Phase budgets are theater without a bandwidth model.** "~1 week" Phase A and "~2 weeks" Phase B are decoration over slip — no FTE, no slack, no critical path. AAR-023 already flagged this shape via my sample finding F-KB-001 and the plan acknowledged it in § 11 ("does not specify the bandwidth model … placeholders") but did not fix it. That's not addressing the finding; that's filing it under "what this plan does NOT do."
- **The acceptance gate (`accept()`) is described as "strict improvement on all dimensions" in D4, but the multi-dimensional ScoreRecord has no specified scalar reduction, no tie-breaking, no slack for noisy dimensions.** Held against the SkillsBench evidence that self-generated skills score −1.3pp on average, a strict-monotone gate over N noisy dimensions will reject nearly every real edit. The acceptance test that the whole product is built around does not have a working specification.
- **§ 13 Step 7 "Hard gate: no `bd claim` until STATUS.md reads RATIFIED" has no enforcement mechanism.** It is honor-system rhetoric. Every other gate in this plan (validate-trilink.sh, partner-name-guard.yml, harness-hash, /validate-consistency) is wired into CI or a hook. This one isn't. The most load-bearing sequencing constraint in the plan is the only one with no failing test behind it.

## Findings

### F-KB-001  [P0]  [SCOPE INTEGRITY]
**Plan section cited:** § 4 Phase A "~1 week budget"; Phase B "~2 weeks budget"; Phase C "~3-4 weeks budget"; § 11 "Does not specify the bandwidth model in calendar weeks beyond rough budgets. Calendar weeks are placeholders…"
**Defect:** Phase budgets are expressed as calendar durations with no FTE assumption, no critical-path identification, no slack budget, no "what work parallelizes vs serializes" decomposition. The plan explicitly punts on this in § 11 — and that punt IS the defect. AAR-023 F-KB-001 (sample finding, but real) raised exactly this shape against the prior draft, and v4.1 responded by adding "see § 4.5 for bandwidth model" pointers to each phase header — but § 4.5 is an Ecosystem Fold *dependency* matrix, not a bandwidth model. The reader cannot tell whether "1 week" assumes the user works alone between truck drives, parallel agent work is presumed, or both. Phase C's "~3-4 weeks" hides a hard `uprg` + `9pi3` dependency that may slip for unbounded duration.
**Why this matters:** Beck canon — "**Slack**" (Beck 2003): *"A plan that does not budget for the things that always go wrong is a plan that always goes wrong."* The plan budgets zero slack for: CI flakes (Beck #1 in AAR-023 cost the j-rig release line two days), npm publish failures (already observed empirically — see CLAUDE.md credentials section's 2026-05-26 NPM_TOKEN lesson), Gemini-review iteration cycles (CLAUDE.md mandates them, plan ignores them), or upstream P0 chain slippage (uprg + 9pi3 close-date is unspecified). First slip cascades silently because there's no model against which to detect it.
**Proposed remediation:** Replace every "~N weeks" with three numbers: (a) optimistic-bead-throughput estimate (e.g., "11 leaf beads × 0.5 day = 5.5 FTE-days"), (b) named critical-path beads, (c) explicit slack budget (Beck's rule of thumb: 1.5× optimistic). Drop calendar weeks entirely. Add a § 4.5 row labeled "BANDWIDTH MODEL" distinct from the Ecosystem Fold matrix.

### F-KB-002  [P0]  [RISK MITIGATION]
**Plan section cited:** Phase A § 4 illustrative API, `declare function accept(scoreV1: ScoreRecord, scoreV2: ScoreRecord): boolean;` + D4 diagram caption "strict improvement on all dims?" + AC-3 "Multi-dimensional score records (Goodhart-resistant). Score is `(skill_hash, eval_set_hash, behavioral_score, readability_score, adversarial_pass_rate, ...)` — never collapsed to a scalar."
**Defect:** The plan asserts (a) the score is multi-dimensional and "never collapsed to a scalar" (AC-3) AND (b) acceptance requires "strict improvement on all dims" (D4 diagram). These two claims, taken together, define an acceptance function that will reject nearly every real edit on a noisy held-out set. There is no specification for: tie-breaking when one dim improves and another stays flat; noise threshold (does +0.01 readability count?); whether `adversarial_pass_rate` is allowed to stay flat at 1.0; what happens when a *new* dim is added to ScoreRecord mid-experiment. Brief pack doc 09 (SkillsBench digest) reports human-curated skills score +16.2pp on average — *with variance across 16/84 model-harness configurations regressing*. A "strict improvement on all dims" gate held against that regression rate will reject ≥16% of true-positive edits. The plan does not name this risk in § 8.
**Why this matters:** Beck canon — "**If you have to test it later, you don't have it yet**" (Beck, TDD by Example, 2002). The acceptance gate IS the product per AC-7 ("the acceptance gate is the durable contribution; the refiner mechanism is swappable"). The whole investment thesis is that this one function is the load-bearing IP. Shipping Phase A with `accept()` as `(v1, v2) => everyDimImproved(v1, v2)` and figuring out the noise model in Phase B is the textbook "we'll get to the test later" anti-pattern. Worse, the Evidence Bundle reports (§ E.1 § 4 + § 5) record accepted-vs-rejected with no statistical framing, so the rejected-edit buffer becomes evidence that the gate is broken, not evidence that the edits were bad.
**Proposed remediation:** Before Phase A claim: write the acceptance predicate as a property test against a synthetic ScoreRecord corpus that includes noise. Specify (a) per-dimension noise tolerance, (b) which dims are strict-monotone vs allowed-to-regress-within-epsilon, (c) tie-breaking, (d) what happens when a new dim is added. Cite the canonical-glossary entry F-LL-001 from the sample findings will produce; the acceptance gate must reference it. The property test is Phase A's first commit, before any propose/score/apply code.

### F-KB-003  [P0]  [SCOPE INTEGRITY]
**Plan section cited:** § 13 Step 7 "**Hard gate:** No `bd claim` against any Skill Refiner bead is permitted until `STATUS.md` reads `RATIFIED`."; README § "Hard gate" repeats the claim.
**Defect:** The Step 7 hard gate has no enforcement mechanism. There is no pre-commit hook that reads `STATUS.md`, no CI job that blocks PR merge on unclaimed bead state, no `bd claim` wrapper that refuses based on file content. Compare to every other gate in this plan: partner-name-guard is a GH Actions workflow; validate-trilink.sh is a CI hook proposed in § 5.5; harness-hash is enforced by audit-harness init; /validate-consistency runs as a pre-commit gate per § 3.5 PR-2. The Step 7 hard gate is enforced by the acting CTO reading the plan and remembering. The most load-bearing sequencing constraint in the entire plan — "do not write code until the audit ratifies" — is the only one without a failing test.
**Why this matters:** Beck canon — "**Make the change easy (warning: this may be hard), then make the easy change**" (Beck, 2012). The "easy change" here is wiring the hard gate into the same enforcement chassis as the other gates: a 20-line bash script in `intent-eval-platform/scripts/check-ratification.sh` that greps `STATUS.md`, paired with a `pre-commit` hook in each of the 5 IEP repos that calls it on commits touching files under any of the Skill Refiner bead clusters. The "hard" change the plan should have made FIRST is exactly that wiring, because shipping the audit without enforcement means Phase 1 review surfaces a P0 finding (this one) that mid-audit becomes a P0 amendment which is the kind of mid-flight retrofit Tidy First exists to prevent.
**Proposed remediation:** Add Step 6.5 (before Step 7): "Wire ratification gate into pre-commit." Implementation: `scripts/check-ratification.sh` reads `audit/2026-05-26-plan-audit/STATUS.md`; if status ≠ `RATIFIED` and the staged diff touches any path matched by the Skill Refiner per-repo cluster (per § 5.5 touch points), exit 1. Add to `.pre-commit-config.yaml` in each of the 5 IEP repos. The honor-system text in Step 7 stays as documentation but the *gate* lives in the chassis.

### F-KB-004  [P1]  [SCOPE INTEGRITY]
**Plan section cited:** § 4 Phase A "Build order (within Epic 1)" table rows 1-8 vs § 4.5 Ecosystem Fold dependencies + Phase A Exit criteria "ship `@j-rig/refiner-core@0.1.0` + `@j-rig/refiner@0.1.0` to npm with sigstore provenance."
**Defect:** Phase A is described as "ships independently" (§ 4 lead paragraph: "Each phase ships independently") yet the Build order step 8 release ceremony depends on the `iep-P3` audit-harness supply-chain hardening (§ 4.5 row: "Phase A `emit-evidence` shell-out, Phase C signed Evidence" — but the *release* itself needs sigstore wired, and `iep-P3` covers the sigstore work). The plan claims Phase A is independent of Phase C and of `uprg`/`9pi3`, but does NOT claim Phase A is independent of `iep-P3` sigstore. Either (a) Phase A ships with un-attested provenance (violates audit-harness norm + erodes the Refiner's own evidence-grade story), or (b) Phase A blocks on the in-flight `iep-P3` chain and the "ships independently" claim is false.
**Why this matters:** Beck canon — "**If you have to test it later, you don't have it yet**" applied to release infrastructure. The exit criterion "ship with sigstore provenance" is a working-end-state claim. The plan does not have a working end-state for Phase A until `iep-P3` is closed, but presents Phase A as the unblocked path while Phases C and beyond block on `uprg`/`9pi3`. Reader cannot tell whether they can start Phase A today or have to wait.
**Proposed remediation:** Either (a) drop the sigstore-provenance requirement from Phase A exit criteria and explicitly state "Phase A ships with un-signed provenance; signed provenance arrives in Phase C via iep-P3 closure" (and accept the optics hit), OR (b) add `iep-P3` as an explicit "→ blocks" row for Phase A in § 4.5 and drop the "ships independently" claim. Pick one. Document the choice in the ratification DR.

### F-KB-005  [P1]  [RISK MITIGATION]
**Plan section cited:** § 8 Risks table row "Frontier moves faster than this plan — Skill Refiner gets eaten by frontier-native skill refinement in 6-12mo"; AC-7 "The acceptance gate is the durable contribution; the refiner mechanism is swappable"; D8 diagram caption "Swap rule: when frontier-native refinement arrives, replace propose() impl in refiner with a thin shim".
**Defect:** Plan names the bitter-lesson risk and asserts the acceptance gate is the durable contribution, but never tests this claim. There is no Phase A spike that runs the eval set against a naive frontier-model-with-in-context-examples baseline (the same null-hypothesis test AAR-023 sample finding F-AK-001 proposed). Without that baseline, the plan cannot tell whether `propose()`'s text-space mutation actually outperforms "just paste the eval set into Opus and ask for a better SKILL.md." If the null hypothesis holds (it might — see AAR-023 brief pack doc 09's SkillsBench finding that effective skills require *human-curated* expertise that models cannot reliably self-generate — but that's about generation, not refinement; refinement is closer to in-context optimization which frontier models DO handle), the entire `propose()` + `apply()` + rejected-buffer infrastructure is scaffolding for a feature the model already provides.
**Why this matters:** Beck canon — "**Three Strikes and You Refactor**" inverted: the plan is refactoring the *future* problem of frontier obsolescence (D8 swap rule) before establishing strike one — i.e., before measuring whether the current mechanism beats the baseline. This is YAGNI for resilience: building the swap-rule architecture before knowing whether the thing being swapped earns its keep.
**Proposed remediation:** Insert Phase A.5 "Null-hypothesis bitter-lesson baseline." Before Phase A step 6 (implement `propose()`), run the bootstrapped eval set against (a) the un-refined skill, (b) the un-refined skill with eval-set-as-in-context-examples appended at runtime, (c) the un-refined skill with the eval set prepended to a single Opus completion asking for a refined SKILL.md. Compare against the (yet-to-be-built) propose() trajectory using the same `accept()` predicate. If lift from (a)→(c) ≥ 70% of projected propose() lift, descope propose() to a thin wrapper around (c) and reallocate Phase A budget to bootstrap() + accept() hardening.

### F-KB-006  [P1]  [RISK MITIGATION]
**Plan section cited:** § 8 Risks table row "Phase C blocks on `uprg`+`9pi3` indefinitely" mitigation: "Phase A + Phase B are NOT blocked on Phase C — they use existing `gate-result/v1` and `SkillSnapshot` + emit via j-rig. Phase C is the polish/first-class-ness layer. Plan can ship A+B without C"
**Defect:** This mitigation is hand-waving. It asserts Phases A+B can ship without C but doesn't specify (a) what the Refiner's Evidence Bundle row looks like in pre-Phase-C output (does it carry a predicate URI? if so which one? `gate-result/v1` is the existing predicate but skill-refiner-pass isn't a gate-result), (b) what happens to A+B-emitted evidence rows when Phase C lands and the predicate URI changes (do they become orphaned? re-emit retroactively? carry a `migration_required: true` flag?), (c) whether the L3 Hook hook's "signed Evidence" gate (D3 diagram) is operational in Phase B if Phase C hasn't shipped the predicate URI. The mitigation says "the plan can ship A+B without C" but doesn't say *what* ships.
**Why this matters:** Beck canon — "**Tidy First?**" (Beck 2024): *"Separate structural changes from behavior changes; small reversible steps that compose."* Shipping Phase B with one predicate URI shape (say, `gate-result/v1` with a `refiner-pass` discriminator) and then changing the shape in Phase C (`skill-refiner-pass/v1` first-class) is a behavior change disguised as a structural change. The Phase B evidence rows are public artifacts (blog post + claude-code-plugins marketplace demo per § 4 Phase B exit criteria) which become irreversible commitments to whatever predicate shape they carry. This is the same Rekor one-way door problem AAR-023 Kleppmann #4 surfaced — the plan filed `uprg` for the Phase C case but didn't extend the discipline to the Phase B case.
**Proposed remediation:** Add to § 8 row a concrete spec: "Pre-Phase-C Refiner evidence rows are emitted as `gate-result/v1` with `gate_id='skill-refiner-pre-tier1'` and a stable JSON shape documented at `intent-eval-lab/specs/skill-refiner-pre-tier1/SHAPE.md`. Phase C migration: the Phase C `skill-refiner-pass/v1` schema MUST be backward-readable against pre-tier1 rows; `uprg` policy (P0) explicitly extends to cover this migration." Without this, Phase B's blog-post demo is a hostage to a Phase C decision that hasn't been made yet.

### F-KB-007  [P2]  [SCOPE INTEGRITY]
**Plan section cited:** § 4 Phase E "Phase E exit criteria: Markdown template committed … HTML renderer (~200 LOC of Hugo template + recharts JS) shipped … One real report generated end-to-end (Phase B exit-criteria demo skill)" + Phase B exit criteria "Phase E Evidence report (markdown form, generated locally) attached to the demo; **public HTML hosting + blog post publication is Phase E's exit criterion, NOT Phase B's** (resolves prior circular Phase B↔E dependency)"
**Defect:** v4.1 claims to have "resolved prior circular Phase B↔E dependency" by moving HTML hosting from Phase B exit to Phase E exit. But Phase E's exit criterion still names "one real report generated end-to-end (Phase B exit-criteria demo skill)" — meaning Phase E cannot exit until Phase B has produced a report. And Phase B's exit ("one real skill refined end-to-end with shipped score-delta evidence") cannot meaningfully demonstrate "evidence" without the template Phase E is supposed to ship. The circularity was renamed, not broken. Phase B ships a "markdown form, generated locally" report against a template that doesn't formally exist until Phase E commits the SPEC.md.
**Why this matters:** Beck canon — "**Make the change easy, then make the easy change.**" The easy change is the template. The hard change being papered over is deciding who owns the template specification. Right now: Phase B uses it, Phase E specifies it, neither phase blocks on the other, and the template lives in superposition. A real working-end-state for Phase B requires the template spec to land first.
**Proposed remediation:** Move the Phase E.1 markdown AAR template specification ("template structure" in § 4 Phase E.1) into a Phase 0 deliverable — commit `intent-eval-lab/specs/skill-refiner-evidence-report/v1.0.0-draft/SPEC.md` BEFORE Phase A claim. Phase B then generates a real report against the real spec. Phase E's remaining work (HTML renderer + Hugo template + one rendered report) becomes a clean follow-on with no circularity. Update § 13 Step 0 to include "commit Phase E template spec" as item 7.
