## TENSION-1 — SkillVersion / SkillSnapshot

> "Bad programmers worry about the code. Good programmers worry about data structures and their relationships."

This is the only tension that actually matters, because it's the only one about the data. Two immutable, content-addressed, parent-pointered entities for what is functionally one concept. That's not "richer state" — that's bad taste.

I don't care what Lamport's state machine says on paper. The kernel ships a Rekor-signed v0.3.0 with this in it, and now you have two ways to express the same fact forever. **Never break userspace.** Pick now or you pay later when downstream consumers have to learn which one means what.

**Arbitration:** merge them. One entity, with a `kind: snapshot | refined | revert | restore` discriminator. Kleppmann's `version_kind` is right; he just doesn't go far enough — collapse the FK too. The architect's "fully orthogonal no-FK" is the second-best answer. Both beat the current plan.

If you can't write three concrete consumer queries that need the two entities distinct at the schema layer — not at the API layer, the _schema_ — then you don't have two entities. You have one entity and a discriminator.

Talk is cheap. Show me the migration SQL.

## TENSION-2 — Phase D

> "Deferred indefinitely at P3" — that's three contradictions in five words.

Karpathy wants quarterly re-measurement triggers. Cunningham wants an anti-goal. Both are correct that the current state is cowardice dressed as planning. "We might do it someday if signals change" is what you write when you can't commit and won't kill.

**Arbitration:** Cunningham wins. Kill it. Make it a Blueprint A § 3 anti-goal. Delete the section from the plan.

If a frontier model release later changes the math, you can un-anti-goal it in 30 minutes with an ISEDC session. Reversibility is cheap; ambiguity is expensive. The plan is already 46 findings deep — don't carry a phase that no one is committed to.

Karpathy's quarterly trigger is fine in theory but it's a calendar event that will get skipped. Process you won't run is decoration.

## TENSION-3 — Process discipline (Cunningham + Hickey)

> "If you have to ask, you don't know how to use it."

Seven parallel process disciplines means six will rot. AC-12 tri-link (CONV-4) is already proof — three writers, no consistency model, "bd-sync as mirror" is the admission that drift IS the steady state.

Cunningham wants to simplify to 1-2. Hickey wants to eliminate AC-12. Both are right; pick whichever ships first.

**Arbitration:** Hickey path. Collapse to **one** source of truth (bd) and treat GH issues + Plane issues as **read-only projections** generated by a one-way sync. No bidirectional `bd-sync`. No "drift detection" CI. If a human edits the GH issue body, the next sync overwrites it. That's the policy. Document it once and stop.

Seven disciplines = no discipline. One enforced sync = a discipline.

## TENSION-4 — Mechanism-is-product vs mechanism-swappable

Not my tension to arbitrate. That's a brand and product-strategy call.

But the engineering observation stands: AC-7 says the mechanism is swappable. If it's swappable, then the mechanism isn't the product — the _interface_ is the product, and you need to (a) define `RefinerStrategy` as a real interface, (b) ship a second implementation that proves the swap works, and (c) stop laundering Karpathy's name on AC-7 until you've actually done the swap.

Pick one: it's the mechanism, or it's the interface. Don't pretend both.

**Arbitration:** define `RefinerStrategy`, ship the second implementation in Phase B, then the AC-7 claim earns its citation. Until then, AC-7 is marketing copy in an acceptance criterion.
